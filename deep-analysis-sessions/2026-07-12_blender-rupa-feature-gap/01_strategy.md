# Analysis Strategy

## Task Structure

The comparison uses Blender 5.1's official manual hierarchy as the source-side
taxonomy and Rupa production source as the target-side evidence. Similar names
do not establish parity. Each overlapping capability is checked across source,
evaluation, selection, UI, Automation/Agent, persistence, diagnostics, and tests.

## Applied Frameworks

- Capability decomposition by Blender's top-level manual areas.
- Route-completeness analysis using Rupa's existing capability gate model.
- Responsibility allocation across exact CAD, universal mesh/procedural modeling,
  DCC modules, domain modules, and adapters.
- Dependency-first prioritization: foundations that unlock multiple capability
  families precede leaf tools.

## Key Layers and Categories

| Layer | Categories |
|---|---|
| Context | Reference scope, comparison method |
| Situation | Blender breadth, Rupa baseline |
| Problem | Modeling foundations, DCC pipeline, production pipeline |
| Issue | Agent reachability, responsibility leakage, completion criteria |
| Solution | Module placement, delivery order |
| Outcome | Auditable gap ledger and implementation focus |

## Evidence Report Design

The report must show:

1. Blender's documented breadth by subsystem.
2. Rupa's current overlap and dominant missing foundations.
3. A visual heat map of partial, internal, and missing capability areas.
4. A dependency-ordered implementation sequence.

## Chartable Data Requirements

- Official manual descendant counts by top-level subsystem.
- Current Rupa operation counts: Swift-CAD feature operations, Agent descriptors,
  and exchange formats.
- Gap state by capability area.
- Priority and ownership by capability block.
